\chapter{Results} \label{chap:results}

The central question addressed in this chapter is whether \ac{FFI} communication can sustain the $20\,\text{Hz}$ execution frequency required by \ac{AC} at simulation scales relevant to \ac{CAS} research. Two complementary evaluations are reported: a controlled benchmark comparing the \ac{FFI} bridge against a socket-based alternative, and a scalability assessment measuring the maximum node population that Collektivity can sustain at real-time frame rates across different hardware configurations and environment complexities.

\section{Comparison with Socket-based Communication} \label{sec:comparison}

An evaluation was conducted to validate the performance of the communication bridge chosen for Collektivity. This study originated from an experimental prototype built with $12$ stationary nodes, a scale chosen to isolate the communication overhead as the sole variable while keeping the collective program's computational cost negligible. The prototype compared two distinct communication strategies: a socket-based backend, in which Unity and Collektive operated as separate processes exchanging data through TCP sockets; and a native backend, in which Collektive was compiled into a shared library and invoked directly by Unity through an \ac{FFI}. While the socket-based approach theoretically offered greater deployment flexibility, it was hypothesized that the native backend would provide the low-latency performance required for high-frequency \ac{AC}.

The results of this benchmark, which has been publicly archived~\footnote{https://github.com/pslab-unibo/experiment-2026-coordination-collektive-unity-benchmark}, revealed a massive performance gap that confirmed the superiority of the native integration. On average, the socket-based backend was found to be over $450$ times slower than the native \ac{FFI} backend (Table~\ref{tab:benchmark}, mean, End-to-end Speedup column). Under peak load, this disparity widened further, with the $99$th-percentile end-to-end slowdown exceeding $747$ times (Table~\ref{tab:benchmark}, p99, End-to-end Speedup column). These findings indicate that while socket-based communication is technically possible, it introduces an `interop tax' that makes real-time simulation of collective systems prohibitive.

The data confirms that the decision to adopt \ac{FFI} was essential for maintaining the $20\,\text{Hz}$ execution frequency required for the simulations described in \cref{chap:casestudy}. By eliminating the overhead of the network stack and inter-process serialization, the \ac{FFI}-based toolchain ensures that the computational budget is spent on aggregate logic and physics interactions rather than on communication bottlenecks.

\section{Benchmark Methodology}

The evaluation of the communication layer was conducted using a controlled experimental prototype featuring $12$ stationary nodes. This scale was deliberately chosen: it is large enough to produce a non-trivial aggregate computation, a distance-based gradient propagated across a connected graph, while remaining small enough to ensure that the measured latency reflects the communication overhead alone, rather than a mixture of communication and computation costs. Whether the same qualitative conclusion holds at 100 nodes is addressed in \cref{sec:scalability}.

To ensure a fair and repeatable comparison, both the socket-based and native test scenarios utilized identical Unity scenes where the communication bridge was the sole variable. The nodes were positioned in a side-by-side linear formation with a designated source node at the center. With a fixed communication radius of 3 units, this topology resulted in a stable connectivity graph where the average distance between neighbors remained consistently 2 units.

The collective program simulated a fundamental \ac{AC} pattern: the discovery and propagation of a distance-based gradient. Each node determined its intensity relative to the central source by identifying the neighbor with the lowest intensity and adding the linear distance to that neighbor. To provide immediate visual feedback, nodes dynamically updated their color in the Unity scene to represent their current gradient value. A strictly synchronous execution model was employed, where Unity halted each frame to await a completed response from the backend. This lock-step synchronization was intentional, as it directly exposed the end-to-end latency of the communication bridge as the primary bottleneck in the simulation loop.

\section{Benchmark Setup and Data Collection}

The benchmarking toolchain was designed for high granularity and statistical significance, tracking at least $1000$ execution cycles for each strategy. The project structure was divided into a Kotlin-based backend and a Unity-based frontend, with data collection points integrated into both. Four distinct datasets were generated: \monospace{native.csv} and \monospace{socket.csv} for backend performance, and \monospace{unity\_native.csv} and \monospace{unity\_socket.csv} for the frontend perspective. To ensure the integrity of the results, a warm-up window of the first $30$ samples was discarded to eliminate the noise associated with initial memory allocations, JIT compilation and connection establishment.

On the backend, the system recorded total service time and the specific duration of the Collektive computation. The distribution of this backend overhead is visualized in~\cref{fig:backend-chart.png}. The y-axis is expressed in nanoseconds; note that the socket distribution extends to approximately $350{,}000\,\text{ns}$ ($0.35\,\text{ms}$) at p95, while the \ac{FFI} distribution remains below $17{,}000\,\text{ns}$ ($0.017\,\text{ms}$) at the same percentile.

\fig{backend-chart.png}{Distribution of backend overhead for FFI and Socket-based communication.}

On the frontend, the end-to-end latency distribution is shown in~\cref{fig:frontend-chart.png}. The y-axis is scaled in units of $10^8\,\text{ns}$, i.e.\ each major gridline represents $100\,\text{ms}$. The socket distribution clusters tightly around $200 \times 10^8\,\text{ns}$ ($200\,\text{ms}$, corresponding to a TCP timeout per round), while the native \ac{FFI} distribution is indistinguishable from zero on this scale, confirming the near-zero latency profile of the bridge.

The entire experiment was packaged into an automated suite. Any researcher with a Linux environment, Python~3.10, and Unity~6000.2.15f1 can replicate the results by running the \monospace{./benchmark.sh} script, which handles process orchestration and chart generation.

\section{Scalability at Simulation Scale} \label{sec:scalability}

The communication benchmark establishes that \ac{FFI} overhead is negligible in isolation. A separate question is whether the complete Collektivity stack (\ac{FFI} bridge, aggregate logic, Unity physics, and 3D rendering combined) can sustain real-time performance at the node populations used in the case study of \cref{chap:casestudy}. \Cref{tab:scalability} summarizes the maximum number of nodes that maintained a frame rate above $30\,\text{FPS}$ on two representative hardware configurations, across both experimental environments.

Both environments reached the same node ceiling on each hardware tier, though at markedly different computational margins. In the minimal environment, the integrated GPU reached 50 nodes with headroom to spare, whereas the rich environment pushed the same hardware to its processing limit at 50 nodes; the dedicated GPU handled 100 nodes comfortably in both cases. The bottleneck in the rich environment is rendering and high-poly mesh collision rather than aggregate logic or \ac{FFI} overhead, which confirms that the communication layer scales correctly and that the performance ceiling is determined by Unity's physics and rendering pipeline, not by Collektivity's integration architecture.
